Wings4U PRD v3.5 v23Topic map: everything the current PRD contains, beyond just features
Updated to anchor to the current PRD structure, workflow model, and document-split plan.
Document identity and scope
Product identity: Wings 4 U online ordering website.
Document purpose: product requirements for customer experience, menu UX, order lifecycle, driver management, admin, operations, and implementation-ready business rules.
Versioning context: current topic map anchored to PRD v3.5 v23, not the older v2.9 snapshot.
Current PRD shape: one heavy master document containing product behavior, workflows, permissions, architecture guidance, data notes, known drawbacks, and backlog.
Customer-facing product features
Public landing page: sticky header, hero, CTAs, trust badges, and footer.
Ordering mode selection: pickup vs delivery, scheduling, address capture, persistent order context bar, and fulfillment-mode revalidation.
Menu experience: categories, search, item cards, availability states, and normalized menu behavior.
Item customization: removals, add-ons, modifiers, validation, and special instructions.
Wings Builder: step-based pound/flavor/saucing workflow with conditional preparation logic.
Wing Combos Builder: combo sizing, side/drink selection, and builder reuse rules.
Cart and checkout: totals, delivery-zone checks, prepayment gate, promo handling, checkout revalidation, and idempotency expectations.
My Orders and tracking: order list, order detail, customer-facing statuses, reorder flow, and live updates.
Reviews and admin replies: item reviews, driver ratings, and reply behavior.
Live order chat: customer/staff conversation tied to active orders only.
Credits, wallet, promos, discounts, and catering inquiry flows.
Operational workflows and lifecycle rules
Order lifecycle: PLACED, ACCEPTED, PREPARING, READY, OUT_FOR_DELIVERY, PICKED_UP, DELIVERED, NO_SHOW_* and CANCELLED.
KDS workflow: single operational KDS surface with shared employee action set and per-status controls.
Cancellation workflow: customer self-cancel window, help flow after cutoff, KDS cancellation request flow, and Admin approval/finalization.
Refund workflow: remaining-refundable-balance trigger, KDS initiation, Admin approval/issuance, and store-credit-first launch behavior.
Driver dispatch workflow: assignment, ETA entry, start delivery, completion, PIN verification, reassignment, overdue handling, and driver-state reset.
Support workflow: issue reporting, ticket lifecycle, resolution options, replacement orders, and auto-ticketing for overdue delivery.
POS and walk-in workflow: WiFi-gated access, order creation, payment method handling, manual discounts, and shared order lifecycle.
Timeclock and register workflows: shifts, breaks, driver availability linkage, register sessions, drawer events, and receipt/print behavior.
Business rules, validation, and edge cases
Selection requirements, quantity/flavor matching rules, and builder-specific validation.
Mode-change revalidation and checkout-time server-side revalidation.
Delivery-zone allowlist, minimum lead time, schedule windows, and minimum delivery subtotal.
No-show counters, prepayment gate, and override logic.
Promo ordering rules, stacking rules, BXGY caveats, and discount precedence.
Wallet/accounting rules, refund constraints, and over-refund/negative-total protections.
Duplicate-order prevention, idempotency, transactional side-effect rules, and locking expectations.
Current edge cases: reorder diffs, unavailable items, sold-out handling, and final-status lock.
Permissions, roles, and access control topics
Two-layer auth model: users.role for coarse actor type and employee_profiles.role for fine-grained staff permissions.
Current KDS model: KDS employees (KITCHEN or MANAGER) handle operational execution, including delivery controls and request/initiation flows defined in Section 7.5.
Admin authority: approval/rejection of cancellation requests, refund approval/issuance, direct admin cancel, catalog/reporting/settings/support/audits.
Cashier and driver role behavior, POS/timeclock access, and future driver-app separation.
Admin bypass rule for staff-role gate, plus continued location scoping and workflow-state enforcement.
Permission source-of-truth rule: canonical permission matrix and KDS role model must drive implementation, not scattered prose.
Multi-location, administration, and business-operations topics
Admin dashboard widgets, reporting, support queues, inventory operations, employee/driver administration, and store settings.
Multi-location architecture: shared database, location-scoped operational data, shared customer accounts, and two-admin setup.
KDS/POS/device/location scoping: printer assignments, WiFi restrictions, location filtering, and no-super-admin warning for current scope.
Operational trade-offs and known concerns: delivery PIN via phone call, manual inventory, add-items timing, KDS request-approval model, WebSocket fallback, overdue-ticketing lag, and static POS IP allowlist.
Endpoint/module split guidance and ownership boundaries across kds, drivers, orders, payments, chat, support, inventory, timeclock, register, reports, and admin_audit.

Test matrix coverage: pickup, delivery, no-show/prepayment, ETA/busy mode, cancel/help, add-items, reviews, chat, driver assignment, ratings, store credit, and promotions.
Acceptance criteria embedded across feature sections and admin sections.
Known drawbacks and design concerns section that records accepted trade-offs instead of hiding them.
Enhancement backlog, non-goals, and staged future items such as driver app, live GPS tracking, saved addresses, and delivery-zone sophistication.

Recommended future document split
PRD: product behavior, UX flows, business rules, permissions, validation, and acceptance criteria.
System Blueprint: architecture, runtime behavior, module boundaries, middleware, devices, print agent, deployment, and scaling concerns.
Schema Spec: tables, fields, enums, constraints, indexes, invariants, and migration triggers.
API Spec: endpoints, request/response contracts, auth rules, error patterns, and event payloads.
Review log / changelog: historical fixes and rationale so the PRD itself stays cleaner.

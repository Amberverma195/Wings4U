2. Ordering Mode Selection
Before showing the menu, the customer must declare their fulfillment mode (Pickup or Delivery), schedule, and (for delivery) their address. This information is locked in for the session and shown persistently in the menu header. Terminology note: the customer-facing UI may use “order type” as a friendly label, but all API payloads, analytics, admin filters, and code must use the canonical field name fulfillment_type (PICKUP | DELIVERY). Never use “order type” in backend code or data — it was deprecated in Section 10.5 because it conflated fulfillment and source. Canonical field mapping for all teams: Display label = “order type” (UI copy only); Canonical field = fulfillment_type (DTOs, DB columns, analytics labels, admin filters). This mapping must be enforced in code review — do not let the friendly display label leak into any data layer.

2.1  Pickup Flow
Triggered by: clicking the "Pickup" CTA on the homepage or changing order mode from the menu header.

Panel / Modal Fields
•  Pickup date selector
•  Pickup time selector (dropdown or time picker)

Rules
•  Minimum lead time: 30 minutes from current time
•  Available time slots must respect store operating hours
•  Default pre-selected time: next available slot (current time + 30 min, rounded to next 5-min interval)
•  On confirm: user proceeds to Menu page with pickup context locked in

2.2  Delivery Flow
Triggered by: clicking the "Delivery" CTA on the homepage or changing order mode from the menu header.

Panel / Modal Fields
•  Delivery address input with autocomplete (Google Places API recommended)
•  Delivery date selector
•  Delivery time selector
•  Contactless delivery preference (Hand to me / Leave at door / Call on arrival / Text on arrival)
•  Buzzer / gate code / unit number field

Rules
•  Address is required before proceeding
•  Minimum lead time: same as pickup (30 minutes) unless updated by owner
•  On confirm: user proceeds to Menu page with delivery context locked in

📌  Delivery Zone — MVP Rule (owner must configure before launch):
  Full delivery zone map enforcement is a near-term feature. For MVP, use a postal code allowlist.
  Admin configures a list of accepted postal codes in location_settings.allowed_postal_codes.
  At address entry: system checks the entered postal code against the allowlist.
  If outside the list: show error "Sorry, we don't currently deliver to that area." before the customer builds a cart.
  Do NOT let the customer reach checkout before discovering the address is not serviceable.
  Owner must provide the initial postal code list before launch. Default: empty list = delivery disabled.

🔄  Cart Revalidation After Mode Change — Required

After any fulfillment_type change (PICKUP → DELIVERY or DELIVERY → PICKUP), the backend must re-run:

  •  Item availability  — some items may be delivery-only or pickup-only
  •  Modifier validity  — modifier groups may differ by fulfillment type
  •  Promo eligibility  — promos have eligible_fulfillment_type; pickup-only promos must be removed when switching to delivery
  •  Delivery fee / tip / prepayment logic  — tip and delivery fee only apply to DELIVERY
  •  Scheduling validity  — the new fulfillment type may have different available time slots

  If any promo, item, or modifier is no longer valid after the mode change:
    →  Remove invalid items and modifiers from the cart server-side, but NEVER silently — the backend must return a structured revalidation diff listing every change: items removed (with reason), modifiers dropped, prices updated. The UI must display this diff clearly to the customer before they proceed, so they know exactly what changed and can confirm or abandon the cart.
    →  Show a banner for removed promos: "Your promo code is not valid for [new mode] orders."
    →  Recalculate totals before showing the updated cart to the customer

  This revalidation must happen server-side. Never trust the frontend to apply it correctly.

•  1 lb  •  1.5 lb  •  2 lb  •  3 lb  •  4 lb  •  5 lb

  •  1 lb / 1.5 lb / 2 lb  →  exactly 1 flavour

Rules:
  •  Slots 2 and 3 only appear when required by weight

7.2  Complete Order Status Lifecycle
📦  PICKUP  —  Full Status Flow
PLACED  →  ACCEPTED  →  PREPARING  →  READY  →  PICKED_UP  (final ✅)
                                                    →  NO_SHOW_PICKUP  (final ❌)
CANCELLED  (final ❌)  —  state machine allows from any non-final state

🛜  DELIVERY  —  Full Status Flow
PLACED  →  ACCEPTED  →  PREPARING  →  READY  →  OUT_FOR_DELIVERY  →  DELIVERED  (final ✅)
                                                                           →  NO_SHOW_DELIVERY  (final ❌)
CANCELLED  (final ❌)  —  state machine allows from any non-final state

🔒  CANCELLED: State Machine vs Permission Authority — These Are Different Things

State machine rule:  CANCELLED is a valid terminal state from any non-final status.
                     The API enforces this. Nothing blocks the transition at the DB level.

Permission rule:     WHO can trigger CANCELLED depends on role + surface:
  •  Customer (CUSTOMER_SELF):      Only within 2-minute window via their own order page
  •  KDS employee cancellation request (KDS_CANCEL_REQUEST):  At PLACED or ACCEPTED status via Reject button, or any non-final status via chat-cancel flow. Requires mandatory reason. Creates a pending cancellation request; Admin must approve before order moves to CANCELLED.
  •  KDS employee via chat flow (KDS_CHAT_REQUEST): From any non-final status. Requires mandatory reason. Creates a pending cancellation request; Admin must approve.
  •  Admin panel direct cancel (ADMIN): From any non-final status. Admin can cancel directly without a pending request. Requires mandatory reason. Admin can also approve/reject pending KDS cancellation requests from this panel.
  ❌  KITCHEN role:                  Cannot directly finalize CANCELLED from any surface. KITCHEN may initiate cancellation requests (Admin approval required to move order to CANCELLED).

Devs must enforce the permission rule at the API layer — not just the state machine.
The state machine says CANCELLED is possible. Permissions say who is allowed to do it.

🔄  Reorder Button — Behaviour Rules

Clicking Reorder attempts to repopulate the cart with the same items from the past order.
The system must validate each item at current state before adding, not blindly clone the snapshot.

Item still available:
  →  Add to cart at current price (NOT snapshot price — snapshots are for historical records, not reorders)

Item unavailable (is_available = false):
  →  Skip the item. Show banner: "Some items are currently unavailable and were not added to your cart: [item name(s)]"

Item soft-deleted (deleted_at is set):
  →  Skip the item. Include in the unavailable banner above.

Item modifiers changed (group or options removed/added):
  →  Add item with still-valid modifiers applied. Removed modifiers are included in the revalidation diff shown to the customer (never dropped silently).
  →  If required modifier group no longer exists: skip the item and include in the banner.
  →  After cart is populated, show a structured reorder diff listing: items skipped (with reason), modifiers dropped, price changes, and required modifier groups missing. Customer must review and confirm or abandon the cart before proceeding to checkout.

Price changed:
  →  Add at current price. No special notice needed — price is always shown before checkout.

If nothing could be added (all items unavailable):
  →  Show: "We're sorry, none of the items from this order are currently available."
  →  Do not open an empty cart or navigate away.

7.5  KDS — Order Ticket Layout & Controls
📌  KDS Surface Architecture — Canonical Model
KDS and Manager Operations belong to one shared operational platform governed by the same permission matrix. The logged-in user’s employee_profiles.role determines which actions are visible and executable. All KDS employees (KITCHEN or MANAGER role): accept, set preparing, set ready, no-show confirmation, ETA adjust, reject/request cancellation with mandatory reason, initiate refund requests, assign drivers, approve busy-driver override, start/complete delivery, and enter delivery PIN. KDS cancellation and refund initiation do not execute directly — they create requests that Admin must approve. Admin Panel: approve/reject cancellation requests, approve/issue refunds, direct cancel of any non-final order, catalog, reporting, settings, support tickets, inventory, employees, drivers, credit issuance, and audits. Every workflow sentence in this PRD that says “KDS employee” means any authenticated KDS user (KITCHEN or MANAGER). The two-tier permission model is: KDS employees handle all operational execution including delivery; Admin handles financial/high-authority approvals (cancellation finalization, refund issuance). For launch, the operational domain may be presented as one interface with role-based controls or as two route groups/workspaces (KDS Workspace for ticket-line execution, prep flow, and ready states; Manager Operations Workspace for exception handling, dispatch, reassignment, overrides, and operational review), provided capabilities remain consistent with the canonical permission model. Manager-specific exception, dispatch, and override workflows may live in a separate Manager Operations workspace without creating a separate authorization model. These are not separate auth systems and not separate role models — they are two operational surfaces inside the same ops platform. Admin Panel = a separate admin-facing management UI outside of the ops platform, used for catalog, settings, reporting, support tickets, inventory, employee management, cancellation approval, refund approval/issuance, and exceptional overrides. Chat-based cancellation requests are available to all KDS employees from the KDS card controls.
KDS Order Card — Required Elements

📌  TOP:  Order type badge (PICKUP — green  |  DELIVERY — orange
📌  Order number + scheduled time (pickup time or delivery time)
📌  Customer name + phone number
📌  Customer stats:  Total Orders: X   |   No-Shows: Y
📌  "Prepayment Required" badge if customer flag is set
📌  Item list with quantities, flavours, saucing method, modifiers, removed ingredients, add-ons
📌  ETA display with 10-second adjustment window on new order arrival (see Section 11)
📌  Action buttons (changes based on current status — see table below)
📌  Print controls: Print Ticket button (triggers local print agent)

⚠️  KDS Cancel Policy — Design Decision (aligned with Section 11 review)

Cancel is NOT exposed as a button on the KDS for operational status updates.
Cancellation from KDS follows an initiate-then-approve workflow. Any KDS employee may request cancellation with mandatory reason capture and full audit logging. The order remains in its current operational status until Admin approves the cancellation request. Admin approval moves the order to CANCELLED. Admin rejection closes the request and the order continues normally. Admin can also directly cancel any non-final order from the Admin panel without a request.

All KDS employees (role: KITCHEN or MANAGER) can:  Accept, Set Preparing, Set Ready, No-Show, Adjust ETA, Request Cancellation with mandatory reason (Admin approval required), Initiate Refund Request (Admin approval required).
Any KDS employee (role: KITCHEN or MANAGER) can request rejection of a new or accepted order with a mandatory reason (see 7.5.1). The rejection request is sent to Admin for approval. The order remains active until Admin approves.
KDS cancellation request is available at PLACED and ACCEPTED status via the Reject button, and at any non-final status via the chat-cancel flow. All KDS-initiated cancellation creates a pending request; only Admin can approve and finalize to CANCELLED.
Full cancellation after PREPARING or later is available via KDS chat-cancel request flow (any KDS employee, mandatory reason, Admin approval required) or directly from the Admin panel.

Why: Direct KDS final cancellation is not allowed because it creates an accountability gap.
      KDS employees may submit cancellation requests with mandatory reason and audit logging. Admin approval is required for final cancellation.
      Mandatory-reason request at PLACED/ACCEPTED keeps operations clean. The request-then-approve model ensures accountability without blocking operational speed.

7.5.1  KDS Order Rejection (PLACED or ACCEPTED)
🚫  KDS Reject Flow — KDS Employee Initiated, Admin Approved

Scenario: order arrives but cannot be fulfilled (out of stock, kitchen closing early, etc.).

1️⃣  KDS employee clicks "Request Cancel" on the KDS card
2️⃣  A modal appears requiring a mandatory reason (free text, min 5 characters)
     Examples: "Out of stock", "Kitchen closing early", "Item no longer available"
3️⃣  KDS employee confirms  →  cancellation_request record created (status = PENDING). Order remains in current status. KDS card shows "Cancel Requested" badge. Admin notified.
4️⃣  Cancellation request record written with:
     •  requested_by_user_id = KDS employee who initiated the request
     •  cancellation_reason = the entered reason
     •  requested_at timestamp
     •  status = PENDING
     On Admin approval, the order record is updated with:
     •  cancelled_by_user_id = Admin who approved the cancellation
     •  cancellation_reason = carried from the cancellation request
     •  cancelled_at = timestamp of Admin approval
     •  cancellation_source = KDS_CANCEL_REQUEST (set when Admin approves the request)
5️⃣  Customer is notified of cancellation with the reason
6️⃣  Admin can see all KDS-rejected orders in Cancelled Orders section
     Filtered view: "Cancellation Requests" shows requesting employee name + role, Admin approver name, reason, request time, approval time, and order details
     This prevents staff rejecting orders because they don't feel like working — full accountability.

🚨  Pickup No-Show — Confirmation Modal
Triggered by clicking "Mark No-Show" when pickup order is in READY status.

Start Delivery

Complete Delivery — with Customer PIN Verification
Driver calls store when trip is done. KDS employee marks DELIVERED in KDS after entering the delivery PIN.
Before marking DELIVERED, the system requires a 4-digit delivery confirmation PIN.

Delivery PIN Flow:
  1️⃣  When order is dispatched (OUT_FOR_DELIVERY), a 4-digit PIN is generated and sent to the customer on live order 
  2️⃣  Customer shows this PIN to the driver at the door
  3️⃣  Driver calls store and relays the PIN to the KDS employee on duty
  4️⃣  KDS employee enters PIN on KDS before clicking "Mark Delivered"
  5️⃣  If PIN matches:  →  status → DELIVERED  →  delivered_at recorded  →  driver state reset (see below)  →  audit event  →  customer UI updates
  6️⃣  If PIN does not match: KDS shows error "Incorrect delivery PIN."
       delivery_pin_verifications.failed_attempts incremented (tracks state, not per-attempt history).
       Each individual failure also logged as a separate admin_audit_logs row (action = PIN_FAIL, order_id, timestamp).
       Retry limits: max 5 failed attempts before PIN entry is locked for this order.
       On lockout: KDS shows "PIN entry locked. Use KDS override to complete delivery."
  7️⃣  PIN bypass: KDS employee can override with mandatory reason (logged in audit)
       KDS employee override available even if PIN is locked; bypass_reason required.
Expired PIN Flow:
  PIN expiry is checked at KDS entry time (compare NOW() against delivery_pin_verifications.expires_at). If the PIN has expired: KDS shows “Delivery PIN has expired.” with two options: (1) Regenerate PIN — system generates a new 4-digit PIN, sends it to the customer, resets expires_at and failed_attempts. Driver must obtain the new PIN from the customer. (2) KDS employee bypass — same flow as PIN mismatch bypass, with mandatory reason logged. A new PIN is NOT automatically sent on expiry — regeneration requires explicit KDS employee action. The customer’s original PIN simply stops working after delivery_pin_expiry_minutes (default 240 min).

Note: When driver app is built, driver enters PIN directly in the app — no phone call needed.

Delivery No-Show / Failed
On confirm:  →  status → NO_SHOW_DELIVERY  →  delivery_no_show_at recorded  →  driver state reset (see below)  →  no-show confirmation flow enforced  →  customer no-show count +1

📌  Canonical Driver State Reset Rule (DELIVERED, NO_SHOW_DELIVERY, or Unassignment):
  is_on_delivery = false
  availability_status = AVAILABLE   (if driver is still clocked in and on shift)
  availability_status = OFF_SHIFT   (if driver has since clocked out)
  Both fields must be updated atomically in the same DB transaction.
  Never update one without the other. State drift between is_on_delivery and availability_status
  will cause the KDS driver picker to show incorrect availability.

(Future — Driver App): Driver receives push notification on assignment, sets their own ETA, enters customer PIN directly, updates delivery status from their device. Changes propagate to Customer UI + KDS + Admin within 3 seconds.

7.8.8  Overdue Delivery — "Almost Arriving" Flow
⏰  Overdue Delivery Detection

Trigger: estimated_arrival_at has passed but order status is still OUT_FOR_DELIVERY.

Customer UI behaviour:
  •  Status label changes from "Out for delivery 🚴" to:
     "Your driver is almost there — hang tight! 😊"
  •  A gentle reassurance message prevents premature customer complaints
  •  No alarm or aggressive messaging — keep the tone calm and positive

Auto support ticket trigger:
  •  If estimated_arrival_at + configurable_grace_period (default: 10 min) has passed:
  •  System automatically creates a support ticket (see Section 25):
     —  ticket_type = DELIVERY_ISSUE
     —  description = "Auto-generated: Delivery significantly overdue. Order #[number]."
     —  created_source = AUTO_OVERDUE
     —  linked to order_id
  •  Admin sees the ticket in support panel
  •  Customer is NOT notified of the auto-ticket — it is an internal alert only
  •  Deduplication rule (non-negotiable): at most ONE auto-generated overdue-delivery ticket per order. Before creating, the background job must check for an existing OPEN, IN_REVIEW, WAITING_ON_CUSTOMER, or RESOLVED ticket where order_id matches and ticket_type = DELIVERY_ISSUE and description starts with “Auto-generated”. If one exists, skip creation. This prevents the 60-second job from spamming duplicate tickets every cycle.
  •  If delivery completes before admin reviews: ticket auto-resolves
       Auto-resolve writes to support_ticket_events:
         event_type = STATUS_CHANGE, from_value = OPEN or IN_REVIEW, to_value = RESOLVED
         performed_by = NULL, note = “Auto-resolved: delivery completed before admin review”
         note = "Auto-resolved: delivery completed before admin review"
       support_ticket_events row written with note = the same message (resolution notes belong on support_ticket_resolutions or support_ticket_events, not as a separate column on support_tickets)

8. Validation Rules & Edge Cases

📌  Menu Item Availability — Precedence Order
When determining if an item can be added to a cart or purchased at checkout, evaluate in this order:

  1️⃣  deleted_at is set (soft-deleted)        →  NOT purchasable. Never shown to customer.
  2️⃣  is_active = false                        →  NOT purchasable. Not shown in menu.
  3️⃣  is_available = false (sold out)          →  NOT purchasable. Shown greyed out with "Sold Out" badge.
  4️⃣  Outside menu_item_schedules window       →  NOT purchasable. Shown greyed with availability hours.
  5️⃣  Not valid for current fulfillment_type   →  NOT purchasable for this mode. Hidden or greyed.
  6️⃣  All checks pass                          →  Purchasable.

Backend must enforce this precedence at checkout, not just at menu display time.
Frontend display is convenience; backend revalidation is enforcement.

🛝  General Checkout Cart Revalidation — Required at Every Checkout Attempt

At checkout (before order creation), the backend must revalidate ALL of the following:

  •  Item availability    — using the precedence rules above
  •  Modifier validity    — required groups still exist; selected options still active
  •  Flavour validity     — selected wing flavours still active; saucing method still valid for count
  •  Promo eligibility   — code still valid, not expired, within usage limits, fulfillment type matches
  •  Scheduling          — selected time slot still within store hours and lead time
  •  Prepayment gate     — if customer has prepayment_required = true, pay-later blocked
  •  Delivery zone       — address postal code still in allowed_postal_codes
  •  Minimum subtotal    — meets delivery minimum if applicable

This is NOT the same as the mode-change revalidation in Section 2.3 (which is mid-session).
This is the final server-side guard at checkout. It catches catalog drift, promo expiry,
  and any state change that happened between cart build and order submission.

If any check fails: return a clear error per failed item/promo. Never silently drop items.
  Customer must see what was removed and why before submitting the order.

10.0  Root Identity Model
One users table is the single source of truth for every person in the system. Role-specific data lives in profile extension tables, not in users itself. This prevents auth and reporting from becoming a mess as the system grows.

Identity Architecture

users                  ← root identity for every person (customer, staff, admin, driver)
  ├─ customer_profiles    ← extends users for customers (loyalty counters, credit balance)
  ├─ employee_profiles    ← extends users for staff (role, PIN, hourly rate, hire date)
  └─ driver_profiles      ← extends employee_profiles for drivers (vehicle, availability, delivery stats)

user_identities        ← login methods per user (PHONE OTP, GOOGLE OAuth)
auth_otp_codes         ← OTP management (or delegated to Firebase / Supabase Auth)

users  (root identity — all roles)
id           UUID PK
role         ENUM: CUSTOMER | STAFF | ADMIN
             -- STAFF is a broad actor type for all restaurant employees.
 -- Authorization granularity (MANAGER | CASHIER | KITCHEN) lives in employee_profiles.role.
 -- See Section 10.0.1 for the canonical auth rule.
is_active    boolean default true
created_at   timestamptz

🔑  Canonical Auth Rule — Two Layers, One Source of Truth Per Layer

Layer 1: users.role  —  coarse actor type for routing and basic access
  •  CUSTOMER   →  customer-facing endpoints only
  •  STAFF      →  any restaurant employee (all staff roles map here at the JWT layer)
  •  ADMIN      →  full admin panel access: approve/reject cancellation requests, approve/issue refunds, direct cancel, catalog, reporting, settings, support tickets, inventory, employees, drivers, credit issuance, audits
  •  DRIVER     →  driver app endpoints only (future)

Layer 2: employee_profiles.role  —  fine-grained staff authorization (STAFF users only)
For KDS permissions, Section 7.5 is canonical. Section 10.0 summarizes identity/auth layering only and must not narrow or override Section 7.5. If any wording below conflicts with the KDS role model in Section 7.5, Section 7.5 wins.
  •  MANAGER  →  all KITCHEN actions (same full KDS operational set), plus approve drawer, edit punches, override prepayment flag. Cannot approve cancellation requests or issue refunds (Admin only).
  •  CASHIER  →  create walk-in orders, apply coupons, clock in/out
  •  KITCHEN  →  full KDS operational actions: accept, reject/request cancel, set preparing, set ready, no-show, ETA adjust, assign driver, approve busy-driver override, start/complete delivery, enter delivery PIN, initiate refund request. Cannot approve cancellation or issue refunds (Admin only).
  •  DRIVER   →  drivers who are also employed staff; same as CASHIER for POS/timeclock access
  ⚠️  ADMIN is NOT in employee_profiles.role. Admin staff use users.role = ADMIN at the JWT layer.
  ⚠️  DRIVER is NOT duplicated: it only appears in employee_profiles when a person is both a driver AND a cashier/kitchen worker.

ADMIN is always permitted on any staff-accessible endpoint, including KDS.
This avoids the contradiction where "Admin has full access" but KDS checks employee_profiles.

Implementation rule:
  if (user.role === ADMIN) { allow; return; }  // bypass staff role check
  // else: check employee_profiles.role for STAFF users

ADMIN users do not need an employee_profiles row. The users.role = ADMIN is sufficient. ADMIN bypasses employee_profiles.role checks only — it does not bypass location scoping (admin_location_assignments), object ownership checks, or workflow-state rules (e.g. final-status lock, cancel permissions). Every admin action is still scoped to their assigned location(s) and still obeys state-machine constraints. To be explicit: “if (user.role === ADMIN) { allow }” applies ONLY to the employee_profiles.role gate. It must NOT be copy-pasted into location-scoping middleware, state-machine validators, or object-ownership checks. Those are separate enforcement layers that ADMIN must still pass. Implementation rule: the ADMIN bypass must live in exactly one middleware — the staff-role gate. Location middleware, state-machine middleware, and ownership middleware must never contain an ADMIN shortcut. If a code review finds “if ADMIN, skip” in any middleware other than the staff-role gate, that is a bug, not a feature.

10.0.1  Guest Checkout Strategy
Decision: Guest checkout creates a users row immediately on order placement.

Rationale:
  •  Every order must have a traceable identity for no-show tracking, prepayment gate, and support tickets.
  •  A temp guest identity that is never converted creates orphaned records and reporting gaps.

Implementation:
  •  Guest provides phone + OTP at checkout (same as registered user phone login).
  •  System checks if a users row exists for that phone via user_identities.
  •  If yes: use the existing user_id. If no: create users row + user_identities row immediately.
  •  Guest is offered account creation (password / Google link) after order confirmation.
  •  All orders placed with the same phone number are attributed to the same user_id.
  •  No-show counters and prepayment flags apply to the user_id, not just the snapshot phone.
📌  Guest Identity Lifecycle — Canonical Rules
Every guest order creates a real users row + user_identities row immediately — there are no temporary, partial, or ghost accounts anywhere in the system. OTP verification at checkout establishes identity trust (is_verified = true on user_identities). Password remains nullable on users until the guest explicitly completes full signup (password set or Google OAuth linked). Role is CUSTOMER from the moment the users row is created — it does not change on signup. If a returning guest places another order with the same phone, the existing user_id is reused — never create a second users row for the same verified phone. After order confirmation, guest is prompted to complete signup (set password / link Google). Declining does not delete or downgrade the account — the user_id persists with full order history, no-show counters, and credit wallet intact.

⚠️  Phone Identity Trade-Off — Documented Limitation

Phone number is the MVP identity key. This is practical for a restaurant context but has known edge cases:

  •  Shared family phones: two people in the same household share one number. Their order history,
       no-show counts, prepayment flags, and credit wallets will merge under one user_id.
  •  Reused numbers: phone numbers are recycled by carriers. A new user with a previously
       used number inherits the history of the original holder.
  •  Business / shared household phones: same merge problem as shared family.

MVP position:
  •  These are accepted trade-offs for MVP. Phone-only identity is simple and frictionless.
  •  Admin can manually override prepayment flags, clear credits, or split accounts on a case-by-case basis.
  •  The system does NOT magically resolve shared phones. It relies on admin discretion for edge cases.

10.5E  Payment Transaction Model
orders.payment_method is a summary field for reporting/UI only. orders.payment_status_summary is a summary field for reporting/UI only (enum: order_payment_status_summary — UNPAID, PENDING, PARTIALLY_PAID, PAID, PARTIALLY_REFUNDED, REFUNDED, FAILED, VOIDED). The canonical source of truth for payment lifecycle is order_payments (append-only transaction rows). Each row represents one money event using payment_tender_method (CASH, CARD, STORE_CREDIT, WAIVED — never SPLIT), transaction_type (AUTH, CAPTURE, VOID, REFUND, ADJUSTMENT), and transaction_status (PENDING, SUCCESS, FAILED, CANCELLED). This supports auth/capture flows, voids, partial refunds, and split payments without hacking the orders row. Canonical rule: order_payments rows are the source of truth for all money movement. orders.payment_method and orders.payment_status_summary are derived convenience fields for reporting and UI display. If orders.payment_method and order_payments rows ever disagree, the rows win. SPLIT as a payment_method value only appears on orders (meaning this order has multiple order_payments rows with different methods). Split payment is not exposed in MVP POS UX, but the backend transaction model supports it via multiple order_payments rows.

Payment Lifecycle Examples

Online card order (Stripe):
  1️⃣  AUTH           →  card pre-authorized at checkout (funds held)
  2️⃣  CAPTURE        →  funds captured when order is ACCEPTED  (not READY — capture on ACCEPTED is the policy)
                    Rationale: ACCEPTED means the kitchen confirmed they will fulfil the order.
                    Capturing later (READY) creates a larger window where the restaurant has committed but the
                    customer is not yet charged, complicating void/refund flows if something goes wrong.
  3️⃣  REFUND         →  partial or full refund if admin resolves issue with refund

Cash walk-in order:
  1️⃣  CAPTURE (CASH) →  recorded immediately when customer pays at counter

Cancelled order (before capture):
  1️⃣  AUTH  →  VOID   →  hold released, customer not charged

Split payment:
  1️⃣  CAPTURE (CARD)         →  partial amount on card
  2️⃣  CAPTURE (STORE_CREDIT) →  remainder from store credit balance
  Each portion creates a separate payments row linked to the same order_id.

10.23  audit_logs Table (named admin_audit_logs in schema)
This table was referenced throughout the PRD but never properly defined. Every critical admin, manager, staff, or system action that changes data must write a row here. Despite the table name admin_audit_logs, it stores actions from all roles — ADMIN, MANAGER, KITCHEN, CASHIER, and SYSTEM. The name is kept for backward compatibility; the intent is a universal compliance trail. admin_audit_logs is the cross-domain compliance trail for sensitive actions. Domain-specific event tables (order_status_events, support_ticket_events, order_eta_events, order_driver_events, order_payments, customer_credit_ledger) remain the source of truth for detailed lifecycle history within their domains. Do not treat admin_audit_logs as a replacement for domain event tables — both are required.
📌  Canonical System-Event Rule
When an action is system-generated (auto-accept, auto-close ticket, auto-resolve overdue ticket, scheduled expiry jobs), set performed_by_user_id = NULL and use the source/event_type field to record SYSTEM as the actor. Do NOT create a fake “system user” row in the users table. NULL performed_by + source = SYSTEM is the canonical pattern across all event tables. Every event table that can receive system-generated writes must have performed_by as nullable.

10.27  Inventory — Current Scope & Future Recipe Mapping
⚠️  Current inventory system is manual-only (MVP)

Admin manually adjusts stock quantities. The system does NOT automatically decrement inventory when an order is placed or completed.

This is intentional for MVP. True stock sync requires:
  •  Recipe / ingredient mappings (e.g. "1 lb wings order uses 1 lb of raw wings + 2 oz sauce")
  •  Unit conversions (e.g. kg → lb → portions)
  •  Waste / yield factors per ingredient
  •  Auto-decrement trigger on order COMPLETED / PICKED_UP / DELIVERED
  •  Low-stock re-check after each auto-decrement

Without recipe mappings, auto-decrement is not meaningful. Build the manual system first, stabilise it, then add recipes as a phase 2 feature.

10.32  Multi-Location — location_id Coverage Audit
Every operational table must include location_id so data can be filtered, reported, and segregated by store. This ensures the shared DB supports multiple locations from day one without schema changes when location 2 opens.

Tables that MUST have location_id (canonical list):

•  orders, order_items (via order), order_status_events (via order)
•  employee_profiles, employee_shifts, employee_breaks
•  driver_profiles, order_driver_events (via order)
•  registers, register_sessions, cash_drawer_events (via register_session)
•  inventory_items, inventory_adjustments (via inventory_item), restock_lists
•  menu_categories, menu_items, modifier_groups
•  location_settings (PK is location_id)
•  refund_requests, support_tickets
•  admin_audit_logs
•  devices (printers, KDS screens, terminals)

Tables that are location-agnostic (do NOT need location_id):
•  users, user_identities, customer_profiles (customers are shared across locations)
•  customer_wallets, customer_credit_ledger
•  promo_codes (promos are configured per code, not per location)
•  catering_inquiries (customer-submitted, no location assignment at submission)

Two-admin setup: the schema is prepared. Each admin gets a row in admin_location_assignments scoping them to their location. No schema changes needed when location 2 opens.

🏗️  Admin Location Scoping — Canonical Model

The previous approach of using employee_profiles.location_id for admin location scoping was a contradiction:
ADMIN was removed from employee_profiles.role, so admins may not have an employee_profiles row at all.
Admin location scoping must NOT depend on employee_profiles.

Solution: admin_location_assignments table
  user_id    FK users(id)       -- must have users.role = ADMIN
  location_id FK locations(id)
  is_primary  boolean default true
  PK: (user_id, location_id)

How it works:
  •  Each ADMIN user gets one row per location they manage (usually just one)
  •  Admin panel endpoint reads admin_location_assignments to scope queries by location_id
  •  Admin sees only data for their assigned location(s)
  •  Two admins for two locations = two users with one assignment row each

This is the source of truth for which admin can see which location data.
It is an explicit FK relationship, not a service-layer convention.

📌  Driver Payout Business Rules

Order attribution (which driver gets paid for which delivery):
  •  Tip and base pay attributed to the driver who COMPLETED the delivery (final assigned_driver_user_id)
  •  If a delivery was reassigned, only the final driver earns tip and base pay for that order
  •  Original driver who was unassigned earns nothing for that order

Order eligibility for a payout batch:
  •  Only orders with final_status = DELIVERED are included in payout batches — full stop.
  •  NO_SHOW_DELIVERY orders are NOT included in payout batches. Driver did not complete the delivery.
  •  If the owner wants to compensate drivers for no-show deliveries, that is a manual adjustment outside this system.
  •  Tips = $0 for NO_SHOW_DELIVERY regardless (no delivery, no tip)
  •  Refunded or disputed orders: tip_cents on order is not modified; payout reflects order data at batch creation time
  •  If an order is refunded AFTER a payout is marked PAID: the payout is NOT retroactively adjusted
      Admin must handle any resulting dispute manually outside the system

Payout batch rules:
  •  One order can only belong to one payout batch per driver — enforced by UNIQUE (order_id, driver_user_id) on driver_payout_order_links
  •  A payout batch can be regenerated (CANCELLED and recreated) before it is marked PAID
  •  Once marked PAID: batch is immutable; corrections require a manual adjustment note outside system
  •  Payout UI tip summary (no batch) can be shown immediately from existing order data
  •  Payout management (batch create, mark paid) requires driver_payouts table rows to exist

12.6  Refund Request Flow (Paid Orders)
If a cancelled order has a remaining refundable balance (net_paid_amount_cents > 0), the system must handle the refund. Refunds are admin-approved. Any KDS employee (KITCHEN or MANAGER) can initiate a refund request from the KDS order detail page. KDS initiation does not issue money — it only opens the refund workflow. Only Admin can approve and issue the refund. No role can issue original-payment refunds directly from KDS.

💳  Refund Request Flow

Trigger: Any cancelled order (by any party) with net_paid_amount_cents > 0 must create or update a refund request workflow for the remaining refundable amount. This covers orders with payment_status_summary = PAID as well as PARTIALLY_REFUNDED orders that still have an outstanding balance.
Remaining refundable amount (computation rule): net_paid_amount_cents is a derived value, not a stored column. Compute as: SUM(signed_amount_cents) from order_payments WHERE transaction_type = CAPTURE AND transaction_status = SUCCESS for the order, PLUS SUM(signed_amount_cents) from order_payments WHERE transaction_type = REFUND AND transaction_status = SUCCESS for the same order (refund rows are negative). Refund automation must be based on this derived balance, not on payment_status_summary alone. payment_status_summary is a summary label that can lag behind the real money state; the order_payments table is the source of truth. The refund request amount_cents for a cancellation-triggered refund must equal this derived remaining balance. See also: financial invariant in Section 10.5E (over-refund guard).
1️⃣  System automatically creates a refund_request record (or updates an existing open request) when a cancelled order has net_paid_amount_cents > 0 (covers PAID and PARTIALLY_REFUNDED orders)
2️⃣  Any KDS employee can also manually initiate a refund request from the KDS order detail page (initiation only — KDS cannot approve or issue refunds)
3️⃣  Admin sees a "Refund Requests" section in the admin panel with:
     •  Order number + customer name + order total + cancellation reason
     •  Who initiated the refund request + when
     •  Payment method used originally
     •  Refund method options (see below)
4️⃣  Admin selects refund method and approves:
     •  Store credit (ACTIVE — available now)
     •  Refund to original payment method (SHOWN but DISABLED for now — will activate with Stripe/payment gateway)
5️⃣  On admin approval:
     •  Store credit:  CREDIT_GRANTED ledger entry → customer_wallets.credit_balance_cents updated
     •  Original method:  order_payments table REFUND entry created → triggers payment gateway call (future)
6️⃣  Refund record stores:
     •  reason, initiated_at, initiated_by_user_id
     •  approved_at, approved_by_user_id
     •  refund_method, amount_cents
     •  status: PENDING | APPROVED | ISSUED | FAILED
     •  notes (admin internal notes)

13. Add Items After Ordering (3-Minute Window)

13.1  How It Works
⚠️  Core Rules

•  Window is 3 minutes after placing (reduced from 5 for operational clarity).
•  System does NOT silently edit the order. All additions go through a tracked request.
•  A change request is created with status PENDING.
•  Admin/Manager can approve or reject (not KDS kitchen staff — see design note below).
•  If order has not yet been ACCEPTED, request may auto-approve if add_items_auto_approve_enabled = true in location_settings (default: false).

📌  Design Note: Approval is admin/manager only.
     KDS is an operational surface for order fulfilment.
     Change request approval is a managerial decision that affects order totals and payment.
     Giving kitchen staff approval power creates billing errors and accountability gaps.
💳  Add-Items Payment Collection — MVP Rule (non-negotiable)
Online card orders: payment is authorized at checkout and captured on ACCEPTED. KDS auto-accept can move an order to ACCEPTED within seconds. After capture, there is no simple way to charge the customer more without a separate payment flow. MVP rule: add-item requests for online card orders are only allowed while the order is still PLACED (before capture). If the order is already ACCEPTED or later, reject the add-item request and show the customer: “Your order is already being prepared. Please place a separate order for additional items.” Cash/POS orders: add-item requests are allowed only within the 3-minute window and only while the order is in a non-final, pre-handoff state (PLACED, ACCEPTED, or PREPARING). Orders in READY, OUT_FOR_DELIVERY, PICKED_UP, DELIVERED, NO_SHOW_CUSTOMER, NO_SHOW_DRIVER, or CANCELLED must reject add-item requests even if the 3-minute window has not elapsed — the updated total is collected at pickup. Store credit orders: add-item delta is deducted from wallet on approval; if wallet balance is insufficient, reject the request. Future: a real delta-charge flow (incremental auth or second charge) can replace this rule once the payment gateway supports it.

Customer — Add Items Flow

4️⃣  Customer sees: "Your request to add items is pending approval."
5️⃣  Approved: items appended, totals updated, customer UI updates
6️⃣  Rejected: customer sees rejection reason (see 13.3) + "Please place a new order."

Chat Read/Unread Behavior (MVP)
Unread state is tracked per side (customer vs staff), not per individual staff member. When a customer sends a message, all staff surfaces (KDS, Manager, Admin) show unread for that order. When any staff member opens the conversation, unread clears for all staff on that thread. When staff sends a message, the customer sees unread on their order detail page. When the customer opens the conversation, customer unread clears. There is no per-staff-member read tracking in MVP, and no user-facing “seen by X” read receipts. Unread state drives the customer-page badge and KDS highlight only. The canonical storage for unread behavior is chat_side_read_states (PK: order_id + reader_side). reader_side is CUSTOMER or STAFF. When any staff member opens the conversation, the service must advance the STAFF-side cursor and clear staff unread for all staff views of that order. chat_read_states (PK: order_id + user_id) is retained as an optional per-user audit/helper layer for analytics and future per-user tracking. 18.3  First-Order Discount
First-Order Discount Rules
•  Applies only to first eligible paid order (not first order placed — to prevent abandoned order abuse)
•  Can be restricted to delivery-only, pickup-only, or both
•  Optional max discount cap and minimum subtotal
•  Controlled as a promo code (is_first_order_only = true) — not a hardcoded rule
•  Admin can disable or change the percentage at any time without code changes

📌  Canonical First-Order Query Rule — must be used everywhere:
  A qualifying prior order = same user_id, AND final_status IN (PICKED_UP, DELIVERED),
  AND net_paid_amount_cents > 0.

  net_paid_amount_cents is a DERIVED value — not a stored column.
  Compute it as: SUM of signed_amount_cents from order_payments table where order_id matches AND transaction_type = CAPTURE AND transaction_status = SUCCESS
                 MINUS SUM of amount_cents from payments where type = REFUND for same order
  Alternatively: check payment_status_summary on orders:
    PAID = net paid > 0 (assume full amount)
    PARTIALLY_REFUNDED = query payments to compute exact net
    REFUNDED = net paid = 0  (disqualifies)
    WAIVED = net paid = 0  (disqualifies)

  Disqualifying cases (order does NOT count as the first eligible paid order):
    •  Cancelled order (including cancelled after capture — payment was reversed)
    •  Fully refunded order (payment_status_summary = REFUNDED — net paid = 0)
    •  Waived order (payment_method = WAIVED)
    •  Partially refunded order where net_paid_cents = 0
    •  POS order with payment_method = STORE_CREDIT and credit was fully reversed

  Qualifying cases (order counts as prior eligible order):
    •  Completed, paid, non-refunded order (PICKED_UP or DELIVERED, payment_status_summary = PAID)
    •  Partially refunded order where net_paid_cents > 0
    •  POS cash order, POS card order, online card order — all count if net paid > 0

18.4  Discount Application Order
When multiple discount types apply, apply in this exact order:

1️⃣  Item-level promotions (BXGY, free item)
2️⃣  Subtotal promotions (% off, fixed amount off)
3️⃣  Delivery fee promotions (free delivery)
4️⃣  Tax calculated on discounted subtotal
5️⃣  Store credit applied (see Section 19)
6️⃣  Final payable total

This order prevents compounding abuse and produces predictable results.
💰  Rounding Policy — Canonical Rule (non-negotiable)
All monetary values are stored as integer cents. No floats anywhere in the money path. Percentage discounts: calculated in decimal, then rounded half-up to the nearest cent (e.g. 10% off $13.57 = $1.357 → $1.36 discount). Tax: calculated on the discounted subtotal after all promos, then rounded half-up to the nearest cent. Rounding level: rounding happens at the order level, not per line item — sum all line amounts in exact cents first, apply percentage to the sum, then round once. This prevents cumulative per-line rounding drift. The backend is the source of truth. If the frontend total differs by even 1 cent, the backend total wins and the frontend must re-sync before confirming payment.
Worked Example 1 — Percentage discount + tax (pickup order)
  3 lb Bone-In Wings           2499
  Poutine                       899
  Drink                         249
  ─────────────────────────────────
  Subtotal (sum in cents)       3647
  10% discount: 3647 × 0.10 = 364.7 → round half-up → 365
  Discounted subtotal:          3282
  HST 13%: 3282 × 0.13 = 426.66 → round half-up → 427
  Grand total:                  3709  ($37.09)
Worked Example 2 — Delivery order with $5 fixed discount + tip + delivery fee
  2 lb Boneless Wings           1999
  Specialty Fries                799
  ─────────────────────────────────
  Subtotal                      2798
  $5.00 fixed discount:        −500
  Discounted subtotal:          2298
  Delivery fee:                 +399
  HST 13% on discounted subtotal: 2298 × 0.13 = 298.74 → round half-up → 299
  Driver tip:                   +300
  Grand total: 2298 + 399 + 299 + 300 = 3296  ($32.96)
Note: tax is calculated on discounted subtotal + delivery fee (delivery fees are taxable for prepared food in Ontario; tax_delivery_fee = true). Tip is NOT taxable (tax_tip = false). These examples are the canonical test cases — frontend, backend, and QA must all produce the same cent values.
✅  TAX CONTRACT — FROZEN FOR LAUNCH
✅  TAX CONTRACT FROZEN — Launch location: London, Ontario. Tax: HST 13% (tax_rate_bps = 1300, tax_snapshot_label = ONTARIO_HST_13). Currency: CAD. Taxable base: discounted_subtotal_cents + delivery_fee_cents (if tax_delivery_fee = true) + driver_tip_cents (if tax_tip = true). Launch defaults: tax_delivery_fee = true (delivery fees ARE taxable for prepared food in Ontario), tax_tip = false (tips NOT taxable), discounts_reduce_taxable_base = true. Formula: tax_amount_cents = round_half_up(taxable_base × tax_rate_bps / 10000). Rounding: round half-up, once, on the total taxable base. Do NOT round per line item. Per-order snapshot: tax_rate_bps, tax_delivery_fee_applied, tax_tip_applied, and tax_snapshot_label are frozen at checkout on every order. Receipts, refunds, and reports MUST use the order-level snapshot, never current config. pricing_snapshot_json stores the full frozen rule inputs for historical reconstruction. Daily tax reporting: daily_tax_summary table (PK: business_date + location_id) stores per-location aggregates: orders_count, taxable_sales_cents, tax_collected_cents, refund_tax_reversed_cents, net_tax_cents. Service updates this table when orders complete or refunds issue. Receipts and refunds still use per-order snapshots; this table is for admin dashboard and reporting only.

Credit application order (at checkout):

  2️⃣  Subtotal promos (% off, fixed amount off)

  •  Card portion          →  refunded to original card second (via payment gateway, future)

  •  Multiple credits: oldest expiry first (FIFO expiry consumption)
  •  Non-expiring credits consumed after all expiring credits

Race condition guard (credit + expiry):
  •  Credit expiry is checked at checkout server-side, not just at display time
  •  If credit expires between cart build and checkout submission: remove from total, show banner
  •  Never allow expired credit to be applied even if UI displayed it as valid

24.6  Device Trust Model
Without a device trust model, any machine on the store WiFi could impersonate a printer terminal. This section defines how devices are authenticated and bound to a location.

🔐  Device Trust Requirements

Device Registration (one-time per device):
  1️⃣  Admin registers a new device from the admin panel: name, type, station_code, location
  2️⃣  System generates a unique API token for that device
  3️⃣  Token is shown once in the UI and never again — admin copies it to the local print agent config
  4️⃣  api_token_hash stored in devices table (bcrypt hash; raw token never stored)
  5️⃣  Device is now registered and active

Device Authentication (every request from print agent):
  •  Local print agent sends its API token as a Bearer token in every request
  •  Backend validates token hash against the devices table
  •  If token does not match a registered active device: 401 Unauthorized
  •  Device must also be is_active = true and location_id must match the order's location_id

Heartbeat (liveness check):
  •  Local print agent pings a heartbeat endpoint every 60 seconds
  •  Backend updates devices.last_seen_at on each ping
  •  If last_seen_at > 5 minutes ago: device is considered offline
  •  Offline device: print jobs queued but not dispatched until device comes back online
  •  KDS shows "Printer offline" badge if assigned printer is offline

Token rotation and revocation:
  •  Admin can rotate a device token from the devices admin page at any time
  •  Rotating generates a new token (shown once); immediately invalidates the old hash in DB
  •  The old print agent receives 401 on its next request and must be reconfigured with the new token
  •  One device has exactly one active token at a time — no multi-token support
  •  If token is suspected leaked: rotate immediately; old token is dead the moment new hash is saved
  •  Deactivating a device (is_active = false) also invalidates its token without rotation

Trust boundary:
  •  WiFi restriction (Section 22.2) limits which machines can reach POS endpoints at the IP layer
  •  Device tokens add a second layer: even on the correct WiFi, unknown machines cannot print
  •  Together: physical location restriction + cryptographic device identity

Schema update: add api_token_hash and device_registered_at to devices table:

devices table additions:
  api_token_hash       text nullable   -- bcrypt hash of the device API token; raw token shown once at registration
  device_registered_at timestamptz nullable   -- when the token was first issued
  token_last_used_at   timestamptz nullable   -- updated on each authenticated request

24.7  Canonical Device Types (Frozen)
Every registered device has exactly one type from this enum. Do not invent synonyms (KITCHEN_TABLET, POS_IPAD, CLOCKIN_TERMINAL, PRINTER_AGENT). If a physical device serves multiple roles, register it once under its primary role and document the secondary role in metadata_json. This enum is frozen and must match the SQL CHECK on devices.device_type.
Canonical enum: KDS_SCREEN | POS_TERMINAL | RECEIPT_PRINTER | CASH_DRAWER | TIMECLOCK_TERMINAL
KDS_SCREEN
Operational kitchen/order-management display. Runs the KDS web app. Receives order queue via WebSocket. Used for: accept/reject orders, status transitions, ETA adjustments, driver assignment, delivery controls, chat, and all operational actions defined in §7.5. Auth: employee session (KITCHEN or MANAGER role). Heartbeat: yes (offline = “KDS offline” alert in admin). Token: device API token for heartbeat; employee auth for actions.
POS_TERMINAL
Order-entry and payment surface for walk-in and phone orders (§22). Runs the POS web app. Used for: creating orders, applying coupons/discounts, collecting payment, printing receipts. Auth: employee session (CASHIER, MANAGER, or ADMIN role) + WiFi IP allowlist (§22.2). Heartbeat: optional but recommended for device health monitoring. Token: device API token for registration; employee PIN for quick-auth.
RECEIPT_PRINTER
Trusted print agent device. Runs the local Node print agent (§24.5). Used for: receipt printing, kitchen ticket printing, drawer-open commands. Auth: device API token (Bearer token on every request). Heartbeat: required (60-second ping; > 5 min stale = offline). Offline behavior: print jobs queued server-side until device reconnects. One printer may serve receipts and kitchen tickets depending on station_key config.
CASH_DRAWER
Cash drawer hardware paired to a register or POS station. Controlled via print agent commands (§24.2). Used for: drawer open on cash sale, no-sale open (manager approval required), safe drop recording. Auth: inherits trust from paired POS/print agent. Not independently authenticated — drawer commands flow through the print agent. Paired to register via station_key.
TIMECLOCK_TERMINAL
Employee punch-in/punch-out surface (§23). Used for: clock in, clock out, break start/end, driver availability sync (clock in = AVAILABLE, clock out = OFF_SHIFT). Auth: device API token + employee PIN for quick-auth + WiFi IP allowlist. Heartbeat: optional. Driver state linkage: timeclock is the MVP source of truth for driver availability (§17). A timeclock device may be a dedicated tablet or share hardware with a POS terminal — the device_type determines which route permissions apply.
Why this matters for trust and auth:
  • Token issuance: each device gets its own API token at registration. One device = one token = one device_type. No sharing.
  • Route permissions: device_type determines which API routes the device can reach. A RECEIPT_PRINTER cannot call KDS status-change endpoints. A TIMECLOCK_TERMINAL cannot create orders.
  • Audit trail: admin_audit_logs and device activity logs reference device_type + device_id so incident response can identify exactly which surface/hardware was involved.
  • Revocation: deactivating or rotating a device token is scoped to exactly one registered device. No cascading breakage across other surfaces.
  • Hardware pairing: station_key links a device to a physical station (e.g. “Front Counter”). CASH_DRAWER and RECEIPT_PRINTER at the same station share a station_key so the system knows which drawer opens when which printer fires.

25. Customer Support & Issue Tickets
Structured post-order complaint and issue tracking system. Replaces ad-hoc DM and phone complaint handling with a trackable workflow.

🔒  Complaint Funnel — Two Paths, Both Explicit

Path 1: Complaint-driven recovery
  Customer issue report  →  support_ticket  →  support_ticket_resolutions  →  (optional) refund_request or credit
  Every complaint from a customer goes through a support_ticket. No exceptions.

Path 2: Cancellation-driven refund (NOT complaint-driven)
  Paid order cancelled  →  refund_request auto-created with support_ticket_id = NULL
  No support ticket is required for a refund triggered by order cancellation.
  This is an operational flow, not a complaint resolution flow.

These two paths are separate and both valid. The schema supports both:
  •  refund_requests.support_ticket_id is nullable (NULL = cancellation-driven, set = ticket-driven)
  •  "Single complaint funnel" means all complaints go through tickets, not that all refunds do.

  The ticket storage and admin resolution system is already fully defined in this section.

25.1  Issue Report — Customer Flow
After an order reaches a final status (DELIVERED or PICKED_UP), customer can report an issue from the order detail page.

•  Wrong item              →  WRONG_ITEM

  DELIVERY_ISSUE covers logistics: late, wrong address, missing item from driver.
  DRIVER_ISSUE covers behaviour: rude, unprofessional, safety concern about the driver specifically.
  Admin routes DRIVER_ISSUE tickets to driver management review. DELIVERY_ISSUE goes to ops/kitchen.

Customer selects issue type + optional description
Optional: attach a photo (future)

25.2  Admin Resolution Flow
Admin can resolve a ticket with:
•  No action   — acknowledged, no compensation
•  Store credit  — triggers Section 19 credit issuance flow
•  Partial refund  — creates refund_request with support_ticket_id set
•  Full refund     — creates refund_request with support_ticket_id set
•  Replacement order  — see rules below
•  Follow-up required  — ticket stays open for further action

Resolution must include a resolution note. Ticket history is preserved — never deleted.

25.2A  Replacement Order — Operational Rules
🔄  Replacement Order — What Happens When Admin Selects "Replacement"

1️⃣  Admin clicks "Create Replacement Order" from the ticket resolution screen
2️⃣  System creates a new order pre-populated with original items, address, and phone. Replacement is a fixed clone of the original order — admin cannot swap items, add extras, or change quantities during creation. If the customer needs different items, admin must issue store credit or a separate comp instead. This prevents value-delta disputes and keeps replacement orders auditable as exact redeliveries.
3️⃣  Order fields:  order_source = ADMIN  |  fulfillment_type = same as original
               payment_method = WAIVED  (replacement is free by default)
               payment_status_summary = PAID (no money collected; $0 order is financially settled)
4️⃣  created_by_user_id = admin who initiated the replacement
5️⃣  support_ticket_resolutions row created with resolution_type = REPLACEMENT and replacement_order_id = new order id
6️⃣  Replacement enters the normal KDS lifecycle (PLACED → ACCEPTED → ...)

Reporting rules for replacement orders:
  •  payment_method = WAIVED  →  counts as $0 revenue in sales reports
  •  Flagged as comp/service-recovery in reporting by filtering: order_source = ADMIN AND payment_method = WAIVED
  •  Can be filtered separately in admin reporting (order_source = ADMIN + payment_method = WAIVED)

Can replacement orders be refunded or credited?
  •  A WAIVED order has no payment to refund.
  •  If the replacement itself is also wrong, admin must open a new support ticket.
  •  Store credit can be issued on top of a replacement if warranted (via credit issuance, not a refund).

25.2B  Resolution Combination Rules
📌  What resolution combinations are allowed on one ticket:

✅  Allowed:
   •  Store credit ONLY
   •  Partial refund ONLY
   •  Full refund ONLY
   •  Replacement ONLY
   •  Follow-up (temporary; ticket stays open)
   •  Store credit + Replacement (admin discretion; two separate actions on the ticket)

❌  Not allowed (prevents double-recovery):
   •  Full refund + Replacement (would comp both money and food)
   •  Full refund + Store credit on the same ticket
   •  Multiple refund_requests for the same full refund amount on one ticket

Closing the financial loop — RESOLVED → CLOSED:
   •  Ticket can move to RESOLVED only when all financial actions are ISSUED (not just APPROVED)
   •  Two valid paths to CLOSED:
        Path A: Admin explicitly marks CLOSED at any time after RESOLVED (no time minimum)
        Path B: Auto-close after support_ticket_autoclose_days if no admin action (see location_settings)
   •  Set support_ticket_autoclose_days = 0 to disable auto-close (Path B off; Path A only)
   •  Auto-close writes support_ticket_events: event_type = STATUS_CHANGE, performed_by = NULL, note = “System auto-close after configured grace period”
   •  Once CLOSED: cannot be reopened. Admin must open a new ticket if the issue recurs.

25.2C  Canonical Support Vocabulary (Frozen)
Support tickets use two separate concepts. Do not invent synonyms, combine them, or create compound states like RESOLVED_REFUND. If this vocabulary drifts across schema, UI, analytics, or filters, reporting will break and nobody will enjoy the resulting archaeology.
Concept A: Ticket Status (lifecycle state — where is this ticket now?)
Canonical enum: OPEN | IN_REVIEW | WAITING_ON_CUSTOMER | RESOLVED | CLOSED
  • OPEN: Ticket received, no staff action taken yet
  • IN_REVIEW: Admin acknowledged, investigation or action underway
  • WAITING_ON_CUSTOMER: Action taken, awaiting customer response or confirmation
  • RESOLVED: All financial actions issued, awaiting close confirmation or grace period
  • CLOSED: Permanently sealed. Cannot reopen. New ticket required.
Lives on: support_tickets.status. This is the current lifecycle state. Source of truth for filtering, queues, and dashboards.
Concept B: Resolution Type (outcome — what action was taken to resolve this ticket?)
Canonical enum: NO_ACTION | STORE_CREDIT | PARTIAL_REFUND | FULL_REFUND | REPLACEMENT | FOLLOW_UP
  • NO_ACTION: Acknowledged, no compensation warranted
  • STORE_CREDIT: Credit issued to customer wallet
  • PARTIAL_REFUND: Partial refund requested/issued
  • FULL_REFUND: Full refund requested/issued
  • REPLACEMENT: Replacement order created (payment_method = WAIVED)
  • FOLLOW_UP: Ticket stays open, further investigation or action required
Lives on: support_ticket_resolutions.resolution_type (one row per resolution action). A ticket can have multiple resolution rows (e.g. STORE_CREDIT + REPLACEMENT). Forbidden combos: FULL_REFUND + REPLACEMENT, FULL_REFUND + STORE_CREDIT on the same ticket.
Summary field on support_tickets (convenience, not source of truth)
support_tickets.resolution_type is a nullable summary convenience field for filtering, badges, dashboards, and quick display (ticket_resolution_type enum — same canonical enum as support_ticket_resolutions). NULL means no resolution has been recorded yet. NO_ACTION is a real business decision, not a placeholder for nothing chosen. It reflects the latest/primary resolution action. Update rule: when a new support_ticket_resolutions row is created, update support_tickets.resolution_type to match. If the summary field and the resolution rows disagree, the rows win. This is the same two-tier pattern used for orders.payment_method vs order_payments (see §10.5E).
Event/history rows (support_ticket_events)
Event rows log transitions and notes: STATUS_CHANGE, RESOLUTION_SET, REOPENED, NOTE_ADDED (PRIORITY_CHANGE retained for forward compatibility). These are history. They do not define new state vocabulary. from_value and to_value are plain text describing what changed, not typed enums. Source of truth for current state = support_tickets.status. Source of truth for resolution detail = support_ticket_resolutions rows. support_tickets.ticket_type is the canonical issue classification field. priority is not a launch-scope field unless explicitly added later. Resolution notes belong on support_ticket_resolutions.note or support_ticket_messages, not as a separate column on support_tickets.

25.3  Ticket Status Flow
OPEN  →  IN_REVIEW  →  WAITING_ON_CUSTOMER  →  RESOLVED  →  CLOSED

📌  Status definitions:
  OPEN:        Ticket received, no action taken yet
  IN_REVIEW: Admin acknowledged, investigation or action underway
  WAITING_ON_CUSTOMER: Action taken, awaiting customer response or confirmation
  RESOLVED:    All financial actions ISSUED (refund/credit/replacement created and fulfilled)
               Ticket can be reopened from RESOLVED if customer reports the issue persists.
  CLOSED:      Admin explicitly marks closed after customer acknowledgment or after grace period.
               Auto-close after X days is configurable per location in location_settings.
               Cannot be reopened once CLOSED — a new ticket must be opened.
Worked Example — Support Ticket Lifecycle (RESOLVED vs CLOSED)
Scenario A — Manual close (admin confirms customer satisfied)
  1. Customer reports WRONG_ITEM on Order #1042        → ticket OPEN
  2. Admin reviews ticket                              → IN_REVIEW
  3. Admin issues $5 store credit (ISSUED)              → RESOLVED
     (ticket stays RESOLVED — customer can still reply)
  4. Admin confirms customer happy, clicks Close        → CLOSED (permanent)
Scenario B — Auto-close (no admin action after resolution)
  1. Customer reports COLD_FOOD on Order #1098          → ticket OPEN
  2. Admin issues replacement order (WAIVED)             → IN_REVIEW
  3. Replacement delivered, all financial actions ISSUED → RESOLVED
  4. 14 days pass, no further action                    → CLOSED (auto)
     support_ticket_events row: performed_by = NULL, note = “System auto-close”
Scenario C — Reopen from RESOLVED (issue persists)
  1. Ticket reaches RESOLVED after store credit issued
  2. Customer replies: “credit was wrong amount”
  3. Admin reopens ticket                               → back to IN_REVIEW
     (allowed from RESOLVED; NOT allowed from CLOSED)
Key distinction: RESOLVED means “we did our part, waiting to confirm customer is satisfied.” CLOSED means “done, permanently sealed.” QA, backend, and admin UI must all treat this boundary consistently.

Priority flag: LOW | MEDIUM | HIGH (admin-set)
Staff can add internal notes (is_internal_note = true, not visible to customer)
Customer-visible replies sent through ticket message thread

📌  Reporting Design Principle
All reports and dashboards below must be built from canonical operational tables, immutable event tables, and order-item snapshots — never from widget-specific or report-shaped tables. Do not create denormalized reporting tables that duplicate the truth stored in orders, order_status_events, payments, customer_credit_ledger, or inventory_adjustments. Report-shaped tables age badly, drift from operational data, and make future product changes harder. If a dashboard needs aggregated data, compute it at query time or via materialized views that refresh from the canonical source.

27. Multi-Location Architecture & Two-Admin Setup
Wings 4 U will expand to multiple locations. The database and schema are designed for this from day one. No schema migrations are needed when location 2 opens.

27.1  Shared Database Strategy
Architecture: One shared Postgres DB, all locations

•  All operational tables carry location_id FK → locations(id)
•  Every query scoped by location_id at the application layer
•  Each location has its own: menu, staff, drivers, register, KDS, inventory, settings
•  Customer accounts are shared (a customer can order from any location)
•  Credit wallets, loyalty (future), and promo codes are global by default (can be scoped per location later)

Opening location 2 requires only:
  1️⃣  INSERT INTO locations → new location row
  2️⃣  INSERT INTO location_settings → prep times, hours, delivery settings
  3️⃣  Create admin user (users.role = ADMIN) + insert admin_location_assignments row for that location
  4️⃣  Create menu, staff, drivers, register for that location
  ✔  No new tables, no migrations, no schema edits required

27.2  Two-Admin Setup
Admin Setup for 2 Locations (Schema Ready Now)

Schema is fully prepared. No changes needed when second admin / location is created.

How it works:
  •  Two rows in users table with role = ADMIN
  •  Each admin linked to their location via admin_location_assignments (user_id, location_id)
  •  Admin panel endpoint reads admin_location_assignments to scope all queries

Data isolation:
  •  Admin A (Location 1) sees: their orders, staff, drivers, inventory, KDS
  •  Admin B (Location 2) sees: their orders, staff, drivers, inventory, KDS
  •  Customer profiles visible to both (customers are shared)

❌  Super-Admin — DO NOT BUILD THIS FROM THIS PRD
  A super-admin who can see all locations is a future need that is NOT designed yet.
  This PRD does not contain a super-admin schema, permission model, or endpoint spec.
  The sentence above only names a possible future approach — it is not a specification.

  To build super-admin you would need, at minimum:
    •  A permission check layer (not in current role model)
    •  A decision on whether users.role gets a SUPER_ADMIN value or this is handled via a flag
    •  Cross-location data scoping rules defined per endpoint
    •  Audit log filtering that does not expose Location 2 data to Location 1 admin accidentally

  For now: two separate location-scoped admins. That is the complete system.

27.3  Driver & Employee Accounts — Multi-Location Ready
No schema changes needed to onboard drivers/employees at location 2:

•  drivers: driver_profiles.location_id assigns them to a home location
•  employees: employee_profiles.location_id assigns their primary location
•  An employee or driver can be reassigned by updating location_id

⚠️  Cross-Location Assignment Design Note:
  employee_profiles uses user_id as PK (one row per user). This means one employee profile per person.
  If a future need arises where a single employee works across multiple locations:
    →  Do NOT create multiple employee_profiles rows for the same user (breaks PK constraint)
    →  Instead, add an employee_location_assignments join table:
         employee_user_id  FK employee_profiles(user_id)
         location_id       FK locations(id)
         is_primary        boolean
         PK: (employee_user_id, location_id)
  For MVP: each employee has one location. The join table is a future-phase addition.

Default driver state:
  •  All drivers created with is_active = true and availability_status = OFF_SHIFT
  •  Driver activates by clocking in at the POS (Section 23.6)
  •  This pattern works identically for location 1 and location 2
  •  Admin can manually override availability_status to AVAILABLE, UNAVAILABLE, or OFF_SHIFT from the driver profile at any time (ON_DELIVERY is system-derived only)

28. Known Drawbacks & Design Concerns
This section documents conscious trade-offs, known limitations, and areas that need re-evaluation before or during implementation. These are not blockers — they are things to be aware of so they do not surprise you mid-build.
📌  Recommended Document Split for Final Deliverables
This PRD carries product behavior, workflows, permissions, architecture guidance, schema reference, known drawbacks, and backlog in one file. That makes it comprehensive but heavy — heavy docs rot because one edit in one section does not always get reflected in every related section. For the implementation phase, split into three living documents: (1) PRD — product behavior, UX flows, acceptance criteria, and the canonical permission matrix. (2) System Blueprint — architecture decisions, module boundaries, runtime behavior, middleware rules, device trust, and deployment notes. (3) Schema Spec — all table definitions, constraints, indexes, enums, and data-model invariants. Each document should cross-reference the others but own its own domain.
